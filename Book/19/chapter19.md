# فصل ۱۹ – عبارات منظم (Regular Expressions)

در چند فصل بعدی، به ابزارهایی خواهیم پرداخت که برای پردازش و دست‌کاری متن به کار می‌روند. همان‌طور که تا اینجا دیدیم، داده‌های متنی نقش بسیار مهمی در همهٔ سیستم‌های شبه‌یونیکس (مثل لینوکس) دارند. اما پیش از آن‌که بتوانیم همهٔ قابلیت‌های این ابزارها را درک کنیم، باید ابتدا با فناوری‌ای آشنا شویم که معمولاً با پیچیده‌ترین کاربردهای این ابزارها همراه است — یعنی **عبارات منظم**.

در مسیر آشنایی‌مان با امکانات و ابزارهای خط فرمان، با ویژگی‌های واقعاً اسرارآمیز زیادی روبه‌رو شدیم؛ از جمله گسترش دستورات (expansion) در شل، نقل‌قول‌ها (quoting)، میانبرهای صفحه‌کلید، تاریخچهٔ دستورات، ویرایشگر vi و موارد دیگر. عبارات منظم هم ادامهٔ همین «سنت» هستند و شاید بتوان گفت از همه‌شان پیچیده‌ترند.
با این حال، این به آن معنا نیست که یادگیری آن‌ها بی‌فایده است — برعکس، درک درست عبارات منظم شما را قادر می‌سازد کارهای حیرت‌انگیزی انجام دهید، هرچند ارزش واقعی‌شان ممکن است در ابتدا آشکار نباشد.

---

### عبارات منظم چیستند؟

به‌طور ساده، **عبارات منظم (Regular Expressions)** نوعی نشانه‌گذاری نمادین هستند که برای شناسایی الگوها در متن استفاده می‌شوند. از برخی جهات، شبیه wildcardهای شل (مثل `*` و `?`) برای تطبیق نام فایل‌ها و مسیرها هستند، اما در مقیاسی بسیار گسترده‌تر.

عبارات منظم توسط بسیاری از ابزارهای خط فرمان و همچنین اکثر زبان‌های برنامه‌نویسی پشتیبانی می‌شوند تا کار با داده‌های متنی را ساده‌تر کنند. البته باید دانست که همهٔ پیاده‌سازی‌های regex دقیقاً یکسان نیستند — نحو (syntax) آن‌ها بسته به ابزار یا زبان اندکی متفاوت است.
در اینجا ما محدود به **استاندارد POSIX** خواهیم بود (که در اکثر ابزارهای خط فرمان استفاده می‌شود) و وارد نسخه‌های پیشرفته‌تر آن در زبان‌هایی مانند Perl نخواهیم شد.

---

### دستور grep

اصلی‌ترین برنامه‌ای که برای کار با عبارات منظم به آن تکیه می‌کنیم، همان دوست قدیمی ماست: `grep`.
نام آن در واقع از عبارت **“global regular expression print”** گرفته شده است؛ یعنی چاپ الگوی منظم سراسری. در اصل، `grep` فایل‌های متنی را می‌گردد تا خطوطی را پیدا کند که با یک عبارت منظم خاص **تطبیق** دارند و آن خطوط را در خروجی چاپ می‌کند.

تا اینجا `grep` را معمولاً برای جستجوی رشته‌های ثابت به کار برده‌ایم، مثلاً:

```bash
ls /usr/bin | grep zip
```

این دستور همهٔ فایل‌هایی را در مسیر `/usr/bin` فهرست می‌کند که در نامشان زیررشتهٔ “zip” وجود دارد.

---

### ساختار دستور

```bash
grep [options] regex [file...]
```

در اینجا `regex` همان عبارت منظم است.
گزینه‌های متداول `grep` به صورت زیر هستند:

| گزینه                             | توضیح                                                                           |
| --------------------------------- | ------------------------------------------------------------------------------- |
| **-i** یا `--ignore-case`         | حساسیت به حروف کوچک و بزرگ را نادیده می‌گیرد.                                   |
| **-v** یا `--invert-match`        | نتیجه را وارونه می‌کند؛ یعنی خطوطی را چاپ می‌کند که **تطبیق ندارند**.           |
| **-c** یا `--count`               | به‌جای چاپ خطوط، فقط تعداد تطبیق‌ها (یا عدم تطبیق‌ها در حالت -v) را چاپ می‌کند. |
| **-l** یا `--files-with-matches`  | فقط نام فایل‌هایی را که تطبیق دارند چاپ می‌کند، نه خود خطوط را.                 |
| **-L** یا `--files-without-match` | برعکس گزینهٔ بالا، فقط نام فایل‌هایی را چاپ می‌کند که هیچ تطبیقی ندارند.        |
| **-n** یا `--line-number`         | شمارهٔ خط را در ابتدای هر خط تطبیق‌یافته چاپ می‌کند.                            |
| **-h** یا `--no-filename`         | در جستجوی چندفایلی، نام فایل را در خروجی حذف می‌کند.                            |

---

### مثال

برای تمرین، چند فایل متنی می‌سازیم:

```bash
ls /bin > dirlist-bin.txt
ls /usr/bin > dirlist-usr-bin.txt
ls /sbin > dirlist-sbin.txt
ls /usr/sbin > dirlist-usr-sbin.txt
ls dirlist*.txt
```

خروجی:

```
dirlist-bin.txt   dirlist-sbin.txt   dirlist-usr-sbin.txt   dirlist-usr-bin.txt
```

حالا جستجو می‌کنیم:

```bash
grep bzip dirlist*.txt
```

خروجی:

```
dirlist-bin.txt:bzip2
dirlist-bin.txt:bzip2recover
```

`grep` در تمام فایل‌های لیست‌شده دنبال رشتهٔ `bzip` می‌گردد و دو تطبیق پیدا می‌کند که هر دو در `dirlist-bin.txt` هستند.

اگر فقط بخواهیم **نام فایل‌هایی** را ببینیم که شامل این رشته‌اند:

```bash
grep -l bzip dirlist*.txt
```

خروجی:

```
dirlist-bin.txt
```

و اگر بخواهیم برعکس، فقط فایل‌هایی را ببینیم که **هیچ تطبیقی ندارند**:

```bash
grep -L bzip dirlist*.txt
```

خروجی:

```
dirlist-sbin.txt
dirlist-usr-bin.txt
dirlist-usr-sbin.txt
```

### نویسه‌های ویژه (Metacharacters) و نویسه‌های معمولی (Literals)

در واقع، تمام جستجوهایی که تاکنون با `grep` انجام داده‌ایم از عبارات منظم استفاده کرده‌اند، فقط از نوع بسیار ساده.
عبارت منظم `"bzip"` یعنی: خطی با حداقل چهار کاراکتر که در آن دنبالهٔ `b`, `z`, `i`, `p` به همان ترتیب بدون فاصله یا نویسهٔ دیگر بینشان وجود دارد.

در اینجا همهٔ نویسه‌ها **عادی (literal)** هستند، چون فقط خودشان را تطبیق می‌دهند.

اما regex می‌تواند شامل **نویسه‌های ویژه یا متاکاراکترها** نیز باشد، که الگوهای پیچیده‌تری را ممکن می‌سازند. متاکاراکترهای اصلی عبارتند از:

```
^  $  .  [ ]  { }  -  ?  *  +  ( )  |  \
```

هر کاراکتر دیگری literal محسوب می‌شود.
کاراکتر `\` (بک‌اسلش) برای ساخت توالی‌های خاص (مانند `\n`، `\t`) یا برای **خنثی‌کردن معنی متاکاراکترها** به کار می‌رود تا آن‌ها به‌صورت نویسهٔ معمولی در نظر گرفته شوند.

> **نکته:**
> بسیاری از این متاکاراکترها در خود شل هم معنا دارند (مثلاً `*`، `$`، `[]`). بنابراین وقتی در خط فرمان از regex استفاده می‌کنید، حتماً باید آن را **داخل کوتیشن ('...') یا ("...")** بگذارید تا شل آن‌ها را گسترش (expand) ندهد.

---

### هر نویسه (The Any Character)

اولین متاکاراکتری که بررسی می‌کنیم، **نقطه (.)** است. این کاراکتر برای **تطبیق با هر نویسه‌ای** در همان موقعیت به کار می‌رود.
اگر آن را در یک عبارت منظم قرار دهیم، هر نویسه‌ای (به‌جز معمولاً کاراکتر پایان خط) را در آن نقطه از رشته تطبیق می‌دهد.

نمونه:

```bash
grep -h '.zip' dirlist*.txt
```

خروجی:

```
bunzip2
bzip2
bzip2recover
gunzip
gzip
funzip
gpg-zip
preunzip
prezip
prezip-bin
unzip
unzipsfx
```

در اینجا عبارت منظم `.zip` به این معناست که **هر نویسه‌ای** به‌عنوان اولین کاراکتر مجاز است، سپس دنبالهٔ `zip` بیاید.
نکته‌های جالبی در نتایج هست:

* برنامهٔ `zip` در خروجی نیست، چون الگوی `.zip` حداقل ۴ نویسه را لازم دارد (هر نویسه + `zip`)، در حالی که کلمهٔ `zip` فقط ۳ نویسه دارد.
* اگر در فهرست فایل‌ها پسوند `.zip` وجود داشت، آن هم تطبیق می‌یافت، چون نقطهٔ واقعی در پسوند فایل نیز به عنوان «هر نویسه» در نظر گرفته می‌شود.

---

### انکرها (Anchors)

کاراکترهای **`^`** و **`$`** در عبارات منظم، **انکر** نامیده می‌شوند.
آن‌ها موقعیت تطبیق را به ابتدای یا انتهای خط محدود می‌کنند:

* `^` یعنی عبارت فقط اگر در **ابتدای خط** باشد تطبیق پیدا کند.
* `$` یعنی عبارت فقط اگر در **انتهای خط** باشد تطبیق پیدا کند.

نمونه:

```bash
grep -h '^zip' dirlist*.txt
```

خروجی:

```
zip
zipcloak
zipgrep
zipinfo
zipnote
zipsplit
```

عبارت `^zip` یعنی تمام خطوطی که با `zip` شروع می‌شوند.

---

```bash
grep -h 'zip$' dirlist*.txt
```

خروجی:

```
gunzip
gzip
funzip
gpg-zip
preunzip
prezip
unzip
zip
```

عبارت `zip$` یعنی تمام خطوطی که با `zip` **پایان می‌یابند**.

---

```bash
grep -h '^zip$' dirlist*.txt
```

خروجی:

```
zip
```

در این حالت عبارت فقط با خطی تطبیق دارد که **فقط شامل واژهٔ zip باشد** (در آغاز و پایان خط).

> نکته: عبارت منظم `^$` (ابتدا و انتهای خط بدون هیچ‌چیز بین آن‌ها) برای یافتن **خطوط خالی** به کار می‌رود.

### یک کاربرد جالب: حل جدول کلمات متقاطع

حتی با همین دانش اندک از regex، می‌توانیم کار مفیدی انجام دهیم.

فرض کنید کسی از شما بپرسد:

> «کلمه‌ای پنج‌حرفی که حرف سومش j و آخرین حرفش r است، چیست؟»

در سیستم لینوکس شما، معمولاً یک **فرهنگ لغت متنی** وجود دارد؛ در مسیر `/usr/share/dict`.
در این پوشه فایل‌هایی مثل `words` وجود دارند که شامل فهرستی از واژه‌ها هستند (هر واژه در یک خط).

مثلاً برای پیدا کردن چنین واژه‌هایی می‌توان نوشت:

```bash
grep -i '^..j.r$' /usr/share/dict/words
```

خروجی:

```
Major
major
```

توضیح الگو:

* `^` → ابتدای خط
* `..` → دو نویسهٔ اول هرچه باشند
* `j` → نویسهٔ سوم باید j باشد
* `.` → هر نویسهٔ چهارم
* `r` → نویسهٔ پنجم باید r باشد
* `$` → انتهای خط

یعنی دقیقاً واژه‌ای پنج‌حرفی با `j` در موقعیت سوم و `r` در آخر.

### براکت‌ها و کلاس‌های نویسه‌ای (Bracket Expressions & Character Classes)

گاهی می‌خواهیم در یک موقعیت خاص، فقط از میان چند نویسهٔ خاص یکی را بپذیریم.
برای این کار از **عبارات براکتی** استفاده می‌کنیم — یعنی نویسه‌ها را درون `[]` می‌گذاریم.

مثال:

```bash
grep -h '[bg]zip' dirlist*.txt
```

خروجی:

```
bzip2
bzip2recover
gzip
```

در اینجا `[bg]zip` یعنی هر خطی که شامل `bzip` یا `gzip` باشد.
یعنی در آن موقعیت، یا `b` یا `g` قابل قبول است.

نکات مهم:

* هر تعداد نویسه را می‌توان داخل براکت گذاشت.
* داخل براکت‌ها، اکثر متاکاراکترها **خاصیت ویژه‌شان را از دست می‌دهند**.
* اما دو استثنا وجود دارد:

  1. **`^`** در ابتدای براکت یعنی **نقیض** (یعنی هیچ‌کدام از این نویسه‌ها نباشد).
  2. **`-`** برای تعریف **بازهٔ نویسه‌ها** استفاده می‌شود (مثلاً `[a-z]` یعنی همهٔ حروف کوچک الفبا).

---

### نفی (Negation)

اگر اولین نویسه در داخل یک عبارت براکتی (bracket expression) علامت **`^`** باشد، یعنی مجموعه‌ای از نویسه‌ها را مشخص می‌کنیم که **نباید** در آن موقعیت وجود داشته باشند.
با این کار می‌توانیم بگوییم: «هر نویسه‌ای به‌جز این‌ها را قبول کن».

نمونه:

```bash
grep -h '[^bg]zip' dirlist*.txt
```

خروجی:

```
bunzip2
gunzip
funzip
gpg-zip
preunzip
prezip
prezip-bin
unzip
unzipsfx
```

در این مثال، تمام خطوطی که شامل `"zip"` هستند و قبل از آن، **هر نویسه‌ای غیر از `b` یا `g`** قرار دارد، نمایش داده شده‌اند.
توجه کنید که فایل `zip` در خروجی نیست، چون در موقعیت قبل از `z` هیچ نویسه‌ای وجود ندارد و negation هنوز **نیاز به وجود یک نویسه دارد**—فقط آن نویسه نباید در مجموعهٔ ردشده‌ها باشد.

> نکته: علامت `^` فقط زمانی معنی نفی دارد که **در ابتدای براکت** باشد؛ در غیر این صورت، یک نویسهٔ عادی محسوب می‌شود.

### بازه‌های سنتی نویسه‌ها (Traditional Character Ranges)

فرض کنید بخواهیم تمام فایل‌هایی را پیدا کنیم که نامشان با **حرف بزرگ** شروع می‌شود:

```bash
grep -h '^[ABCDEFGHIJKLMNOPQRSTUVWXZY]' dirlist*.txt
```

این کار می‌کند، ولی تایپ کردن همهٔ حروف انگلیسی آزاردهنده است.
بنابراین از یک **بازهٔ نویسه‌ای** استفاده می‌کنیم:

```bash
grep -h '^[A-Z]' dirlist*.txt
```

خروجی:

```
MAKEDEV
ControlPanel
GET
HEAD
POST
X
X11
Xorg
MAKEFLOPPIES
NetworkManager
NetworkManagerDispatcher
```

در اینجا `[A-Z]` یعنی هر نویسه‌ای بین `A` تا `Z`.
می‌توان چند بازه را هم ترکیب کرد:

```bash
grep -h '^[A-Za-z0-9]' dirlist*.txt
```

این یکی همهٔ خطوطی را که با **حرف یا عدد** شروع می‌شوند تطبیق می‌دهد.

---

### استفاده از خط تیره و جایگاه خاص آن

کاراکتر **`-`** در داخل براکت‌ها برای مشخص‌کردن بازه‌ها به کار می‌رود، ولی اگر بخواهیم **خود کاراکتر خط تیره را** به عنوان نویسهٔ معمولی بنویسیم، باید آن را **در ابتدای براکت** قرار دهیم:

مثلاً:

```bash
grep -h '[A-Z]' dirlist*.txt
```

→ تطبیق هر نامی که دارای یک حرف بزرگ باشد.

اما:

```bash
grep -h '[-AZ]' dirlist*.txt
```

→ تطبیق هر نامی که شامل یکی از این‌ها باشد: خط تیره (`-`)، حرف بزرگ `A`، یا `Z`.

### کلاس‌های نویسه‌ای در POSIX (POSIX Character Classes)

بازه‌های سنتی راهی ساده و قابل فهم برای تعریف مجموعهٔ نویسه‌ها هستند، اما همیشه درست کار نمی‌کنند.
علت این مشکل به **ترتیب چینش نویسه‌ها (collation order)** در سیستم‌های مختلف برمی‌گردد.

در گذشته، یونیکس فقط با **کدگذاری ASCII** کار می‌کرد. ترتیب آن چنین بود:

```
ABCDEFGHIJKLMNOPQRSTUVWXYZabcdefghijklmnopqrstuvwxyz
```

اما ترتیب «فرهنگی» (dictionary order) به این صورت است:

```
aAbBcCdDeEfFgGhHiIjJkKlLmMnNoOpPqQrRsStTuUvVwWxXyYzZ
```

بعدها با گسترش زبان‌ها، مجموعه‌نویسه‌ها به UTF-8 و locale وابسته شدند. مثلاً:

```bash
echo $LANG
```

خروجی نمونه:

```
en_US.UTF-8
```

در این حالت برنامه‌های سازگار با POSIX از ترتیب دیکشنری استفاده می‌کنند، نه ASCII، و همین باعث می‌شود که مثلاً `[A-Z]` به شکل متفاوتی تفسیر شود (در برخی localeها حتی حروف کوچک را هم شامل می‌شود، جز `a`).

برای رفع این مشکل، استاندارد POSIX مجموعه‌ای از **کلاس‌های نویسه‌ای** تعریف کرد که از locale مستقل‌اند و به شکل زیر نوشته می‌شوند:

```
[:classname:]
```

و همیشه داخل براکت دوم قرار می‌گیرند، مثلاً `[[:digit:]]` یا `[[:alpha:]]`.

---

### جدول کلاس‌های نویسه‌ای POSIX

| کلاس نویسه   | توضیح                                         | معادل تقریبی در ASCII            |      |
| ------------ | --------------------------------------------- | -------------------------------- | ---- |
| `[:alnum:]`  | حروف و ارقام الفبایی                          | `[A-Za-z0-9]`                    |      |
| `[:word:]`   | مانند alnum اما به‌علاوه‌ی زیرخط `_`          | `[A-Za-z0-9_]`                   |      |
| `[:alpha:]`  | فقط حروف الفبا                                | `[A-Za-z]`                       |      |
| `[:blank:]`  | فاصله و تب (space, tab)                       | `[ \t]`                          |      |
| `[:cntrl:]`  | نویسه‌های کنترلی (کدهای ۰ تا ۳۱ و ۱۲۷)        | —                                |      |
| `[:digit:]`  | ارقام ۰ تا ۹                                  | `[0-9]`                          |      |
| `[:graph:]`  | نویسه‌های قابل‌مشاهده (کدهای ۳۳ تا ۱۲۶)       | —                                |      |
| `[:lower:]`  | حروف کوچک                                     | `[a-z]`                          |      |
| `[:punct:]`  | علائم نگارشی                                  | `[-!"#$%&'()*+,./:;<=>?@[\\\]_`{ | }~]` |
| `[:print:]`  | نویسه‌های قابل چاپ + فاصله                    | `[[:graph:] ]`                   |      |
| `[:space:]`  | نویسه‌های فاصله‌ای (space, tab، newline، ...) | `[ \t\r\n\v\f]`                  |      |
| `[:upper:]`  | حروف بزرگ                                     | `[A-Z]`                          |      |
| `[:xdigit:]` | نویسه‌های عددی شانزده‌هشتی (۰–۹، A–F، a–f)    | `[0-9A-Fa-f]`                    |      |

---

مثال:

```bash
ls /usr/sbin/[[:upper:]]*
```

خروجی نمونه:

```
/usr/sbin/MAKEFLOPPIES
/usr/sbin/NetworkManagerDispatcher
/usr/sbin/NetworkManager
```

این همان نتیجهٔ `[A-Z]*` را می‌دهد، اما با رفتار قابل پیش‌بینی در تمام localeها.

> توجه:
> در این مثال دستور `ls` در حال انجام **گسترش مسیرها (pathname expansion)** است، نه اجرای regex.
> با این حال، همین کلاس‌های نویسه‌ای را می‌توان در regex نیز به کار برد.

### بازگشت به ترتیب سنتی (ASCII) در مقایسه نویسه‌ها

(**Reverting to Traditional Collation Order**)

اگر بخواهید سیستم‌تان به‌جای ترتیب لغت‌نامه‌ای (dictionary order) از **ترتیب سنتی ASCII** برای مقایسه و مرتب‌سازی نویسه‌ها استفاده کند، کافی است مقدار متغیر محیطی `LANG` را تغییر دهید.
همان‌طور که دیدیم، متغیر `LANG` تعیین می‌کند که زبان و مجموعه‌نویسهٔ locale شما چیست — مقداری که در هنگام نصب لینوکس، بر اساس زبان انتخابی شما تنظیم شده است.

برای دیدن تنظیمات فعلی locale، دستور زیر را اجرا کنید:

```bash
locale
```

خروجی نمونه:

```
LANG=en_US.UTF-8
LC_CTYPE="en_US.UTF-8"
LC_NUMERIC="en_US.UTF-8"
LC_TIME="en_US.UTF-8"
LC_COLLATE="en_US.UTF-8"
LC_MONETARY="en_US.UTF-8"
LC_MESSAGES="en_US.UTF-8"
LC_PAPER="en_US.UTF-8"
LC_NAME="en_US.UTF-8"
LC_ADDRESS="en_US.UTF-8"
LC_TELEPHONE="en_US.UTF-8"
LC_MEASUREMENT="en_US.UTF-8"
LC_IDENTIFICATION="en_US.UTF-8"
LC_ALL=
```

برای بازگرداندن رفتار سنتی یونیکس، مقدار `LANG` را به `POSIX` تغییر دهید:

```bash
export LANG=POSIX
```

با این کار، سیستم از **زبان انگلیسی آمریکا با کدگذاری ASCII** استفاده می‌کند.
قبل از انجام این تغییر، مطمئن شوید که واقعاً می‌خواهید چنین رفتاری داشته باشید، چون ممکن است حروف غیرانگلیسی (مثلاً فارسی) به‌درستی نمایش داده نشوند.

برای دائمی کردن این تغییر، می‌توانید این خط را به انتهای فایل `~/.bashrc` خود اضافه کنید:

```bash
export LANG=POSIX
```

### عبارات منظم پایه و پیشرفته در POSIX

(**POSIX Basic vs. Extended Regular Expressions**)

درست وقتی فکر می‌کنیم ماجرا نمی‌تواند پیچیده‌تر شود، متوجه می‌شویم که استاندارد POSIX خودش عبارات منظم را به دو نوع تقسیم کرده است:

* **BRE** → Basic Regular Expressions (عبارات منظم پایه)
* **ERE** → Extended Regular Expressions (عبارات منظم پیشرفته)

هر برنامه‌ای که با استاندارد POSIX سازگار است و از BRE پشتیبانی می‌کند، باید ویژگی‌هایی را که تاکنون گفتیم داشته باشد.
برنامهٔ `grep` نمونه‌ای از همین ابزارهاست.

---

#### تفاوت اصلی بین BRE و ERE

در **BRE**، فقط متاکاراکترهای زیر معنا دارند:

```
^  $  .  [ ]  *
```

تمام نویسه‌های دیگر به‌عنوان **حروف عادی (literal)** در نظر گرفته می‌شوند.

اما در **ERE**، متاکاراکترهای بیشتری پشتیبانی می‌شوند:

```
( )  { }  ?  +  |
```

نکتهٔ گیج‌کننده اینجاست:
در BRE، اگر بخواهید از `(` یا `{` به‌عنوان متاکاراکتر استفاده کنید، باید آن‌ها را **با بک‌اسلش (`\`) فرار دهید**.
ولی در ERE، برعکس است — اگر متاکاراکتری را با `\` بنویسید، **دیگر literal محسوب می‌شود**.

به‌عبارت دیگر:

| کاراکتر | در BRE                           | در ERE                      |
| ------- | -------------------------------- | --------------------------- |
| `(`     | فقط اگر `\(` باشد متاکاراکتر است | بدون بک‌اسلش متاکاراکتر است |
| `{`     | فقط اگر `\{` باشد متاکاراکتر است | بدون بک‌اسلش متاکاراکتر است |

بنابراین در BRE و ERE دقیقاً برعکس هم عمل می‌کنند.

---

#### استفاده از ERE در `grep`

در نسخه‌های قدیمی یونیکس، برای استفاده از ERE از برنامهٔ جداگانه‌ای به نام `egrep` استفاده می‌شد.
اما در نسخهٔ GNU از `grep`، نیازی به برنامهٔ جدا نیست — کافی است گزینهٔ `-E` را اضافه کنید:

```bash
grep -E 'expr'
```

که معادل اجرای:

```bash
egrep 'expr'
```

است.

---

### پیش‌زمینه‌ای کوتاه دربارهٔ POSIX

در دههٔ ۱۹۸۰، یونیکس به‌شدت در میان شرکت‌های سخت‌افزاری محبوب شد.
شرکت‌های مختلف کد منبع سیستم‌عامل یونیکس را از **AT&T** می‌خریدند و نسخه‌های خاص خودشان را توسعه می‌دادند.
اما هر شرکت برای متمایز شدن، تغییرات اختصاصی (proprietary extensions) اضافه می‌کرد، و همین باعث شد نرم‌افزارها دیگر بین نسخه‌ها **سازگار نباشند**.
این دوران تاریک در تاریخ یونیکس به‌نام **"Balkanization"** شناخته می‌شود — یعنی تکه‌تکه شدن.

برای حل این مشکل، مؤسسهٔ **IEEE (Institute of Electrical and Electronics Engineers)** وارد عمل شد و در اواسط دههٔ ۸۰ میلادی، استانداردهایی برای تعیین رفتار یکنواخت سیستم‌های شبه‌یونیکس تدوین کرد.
این استانداردها با نام رسمی **IEEE 1003** شناخته شدند و شامل تعریف APIها، شل، و ابزارهایی بودند که باید در هر سیستم یونیکس‌مانند وجود داشته باشند.

نام **POSIX** از عبارت *Portable Operating System Interface* گرفته شده است (و حرف X در پایان برای جذاب‌تر شدن اضافه شده).
پیشنهاد این نام را **ریچارد استالمن** (بله، همان بنیان‌گذار جنبش نرم‌افزار آزاد) داده بود و توسط IEEE پذیرفته شد.

### جایگزینی (Alternation)

اولین ویژگی از **عبارات منظم پیشرفته (Extended Regular Expressions)** که بررسی می‌کنیم، مفهوم **alternation** است — یعنی توانایی انتخاب تطبیق از میان چند الگو.

به‌طور مشابه که در براکت‌ها (`[ ]`) می‌توان از میان چند نویسه یکی را انتخاب کرد، با alternation می‌توان از میان چند **عبارت یا رشته** یکی را انتخاب کرد.

---

#### مثال ساده

```bash
echo "AAA" | grep AAA
```

خروجی:

```
AAA
```

اما اگر چیزی متفاوت بفرستیم:

```bash
echo "BBB" | grep AAA
```

خروجی خالی است، چون تطبیقی وجود ندارد.

---

#### استفاده از alternation با متاکاراکتر `|`

```bash
echo "AAA" | grep -E 'AAA|BBB'
```

خروجی:

```
AAA
```

و اگر مقدار دیگر را امتحان کنیم:

```bash
echo "BBB" | grep -E 'AAA|BBB'
```

خروجی:

```
BBB
```

اما:

```bash
echo "CCC" | grep -E 'AAA|BBB'
```

خروجی ندارد.

عبارت `'AAA|BBB'` یعنی «رشته‌ای را تطبیق بده که یا AAA باشد یا BBB».
چون از ویژگی‌های ERE استفاده کرده‌ایم، باید گزینهٔ `-E` را به `grep` اضافه کنیم
(یا از `egrep` استفاده کنیم). همچنین regex را داخل `' '` قرار می‌دهیم تا `|` به‌جای **pipe** تفسیر نشود.

می‌توان بیش از دو گزینه نیز داشت:

```bash
echo "AAA" | grep -E 'AAA|BBB|CCC'
```

---

#### ترکیب alternation با سایر الگوها

برای جلوگیری از ابهام، می‌توان بخش‌های جایگزین را با پرانتز جدا کرد:

```bash
grep -Eh '^(bz|gz|zip)' dirlist*.txt
```

الگوی بالا خطوطی را پیدا می‌کند که با `bz` یا `gz` یا `zip` شروع می‌شوند.

اما اگر پرانتزها را حذف کنیم:

```bash
grep -Eh '^bz|gz|zip' dirlist*.txt
```

معنی تغییر می‌کند؛ حالا این الگو خطوطی را تطبیق می‌دهد که
یا با `bz` شروع می‌شوند **یا** شامل `gz` هستند **یا** شامل `zip`.

### کمیت‌سنج‌ها (Quantifiers)

عبارات منظم پیشرفته به ما اجازه می‌دهند **تعداد تکرار الگوها** را نیز مشخص کنیم.
چند متاکاراکتر برای این کار استفاده می‌شوند: `?`, `*`, `+`, `{}`
در این بخش به سه مورد اول می‌پردازیم.

---

#### `?` — تطبیق یک عنصر صفر یا یک‌بار

علامت سؤال یعنی «وجود این بخش اختیاری است» — یعنی ممکن است صفر بار یا یک‌بار وجود داشته باشد.

مثلاً فرض کنید بخواهیم صحت شماره تلفن را بررسی کنیم و دو حالت مجاز باشند:

```
(555) 123-4567
555 123-4567
```

در regex به این صورت تعریف می‌شود:

```bash
^\(?[0-9][0-9][0-9]\)? [0-9][0-9][0-9]-[0-9][0-9][0-9][0-9]$
```

توضیح:

* `\(?` و `\)?` → پرانتز اختیاری (صفر یا یک‌بار)
* `[0-9]` → عدد از ۰ تا ۹
* فاصله و خط تیره همان‌گونه که هستند نوشته می‌شوند
* `^` و `$` → ابتدای و انتهای خط برای محدودکردن تطبیق کل خط

---

#### آزمایش

```bash
echo "(555) 123-4567" | grep -E '^\(?[0-9][0-9][0-9]\)? [0-9][0-9][0-9]-[0-9][0-9][0-9][0-9]$'
```

خروجی:

```
(555) 123-4567
```

---

```bash
echo "555 123-4567" | grep -E '^\(?[0-9][0-9][0-9]\)? [0-9][0-9][0-9]-[0-9][0-9][0-9][0-9]$'
```

خروجی:

```
555 123-4567
```

---

```bash
echo "AAA 123-4567" | grep -E '^\(?[0-9][0-9][0-9]\)? [0-9][0-9][0-9]-[0-9][0-9][0-9][0-9]$'
```

خروجی ندارد.
یعنی فقط شماره‌هایی با ۱۰ رقم معتبر هستند.

---

#### `*` — تطبیق صفر یا چند بار

علامت ستاره `*` مشابه `?` است، با این تفاوت که اجازه می‌دهد یک بخش **صفر بار یا هر تعداد بار تکرار شود**.

مثلاً می‌خواهیم بررسی کنیم که یک رشته **شبیه جمله** است —
یعنی با حرف بزرگ شروع شود، سپس شامل هر تعداد حروف (بزرگ یا کوچک) و فاصله باشد و در نهایت با نقطه تمام شود.

الگو:

```bash
[[:upper:]][[:upper:][:lower:] ]*\.
```

اجزای عبارت:

1. `[[:upper:]]` → اولین حرف باید بزرگ باشد.
2. `[[:upper:][:lower:] ]*` → سپس هر تعداد حرف (کوچک یا بزرگ) یا فاصله مجاز است.
3. `\.` → باید با نقطه پایان یابد (نقطهٔ literal).

---

#### آزمایش

```bash
echo "This works." | grep -E '[[:upper:]][[:upper:][:lower:] ]*\.'
```

خروجی:

```
This works.
```

---

```bash
echo "This Works." | grep -E '[[:upper:]][[:upper:][:lower:] ]*\.'
```

خروجی:

```
This Works.
```

---

```bash
echo "this does not" | grep -E '[[:upper:]][[:upper:][:lower:] ]*\.'
```

خروجی ندارد، چون با حرف کوچک شروع شده و نقطه هم ندارد.

---

نتیجه:
الگوی بالا فقط جمله‌هایی را تطبیق می‌دهد که:

* با حرف بزرگ شروع شوند،
* در میانشان فقط حروف و فاصله باشند،
* و با نقطه تمام شوند.

---

### `+` — تطبیق یک عنصر **یک یا چند بار**

متاکاراکتر `+` شبیه `*` عمل می‌کند، با این تفاوت که حداقل **باید یک بار** از عنصر قبلی وجود داشته باشد تا تطبیق انجام شود.
در حالی که `*` صفر بار را هم مجاز می‌داند، `+` می‌گوید «حداقل یک بار باید تکرار شود».

به‌عنوان مثال، فرض کنید می‌خواهیم خطی را تطبیق دهیم که فقط شامل گروه‌هایی از **حروف الفبا** باشد که با **فاصلهٔ تکی** از هم جدا شده‌اند:

```bash
^([[:alpha:]]+ ?)+$
```

توضیح:

* `[[:alpha:]]+` → یک یا چند حرف الفبا (A–Z یا a–z)
* `?` → یک فاصله (اختیاری)
* پرانتزها گروه را مشخص می‌کنند و `+` در انتها یعنی این گروه می‌تواند چند بار تکرار شود.
* `^` و `$` تضمین می‌کنند که کل خط فقط از همین ساختار تشکیل شده است.

---

#### آزمایش

```bash
echo "This that" | grep -E '^([[:alpha:]]+ ?)+$'
```

خروجی:

```
This that
```

```bash
echo "a b c" | grep -E '^([[:alpha:]]+ ?)+$'
```

خروجی:

```
a b c
```

```bash
echo "a b 9" | grep -E '^([[:alpha:]]+ ?)+$'
```

خروجی ندارد، چون در خط نویسه‌ای غیراحرفی (`9`) وجود دارد.

```bash
echo "abc  d" | grep -E '^([[:alpha:]]+ ?)+$'
```

خروجی ندارد، چون بین `c` و `d` دو فاصله است (الگو فقط یک فاصلهٔ تکی مجاز می‌داند).

---

### `{ }` — تطبیق تعداد **مشخصی** از تکرارها

متاکاراکترهای `{` و `}` برای تعیین **تعداد دقیق یا بازه‌ای** از تکرار یک عنصر استفاده می‌شوند.
چهار حالت کلی برای استفاده از آن‌ها وجود دارد:

| قالب    | معنا                                                   |
| ------- | ------------------------------------------------------ |
| `{n}`   | عنصر باید دقیقاً `n` بار تکرار شود.                    |
| `{n,m}` | عنصر باید حداقل `n` بار و حداکثر `m` بار تکرار شود.    |
| `{n,}`  | عنصر باید حداقل `n` بار تکرار شود (بدون محدودیت بالا). |
| `{,m}`  | عنصر باید حداکثر `m` بار تکرار شود.                    |

---

#### مثال: بازنویسی عبارت شماره تلفن

در مثال قبلی برای شماره تلفن از این الگو استفاده کردیم:

```bash
^\(?[0-9][0-9][0-9]\)? [0-9][0-9][0-9]-[0-9][0-9][0-9][0-9]$
```

با استفاده از `{}` می‌توانیم آن را ساده‌تر کنیم:

```bash
^\(?[0-9]{3}\)? [0-9]{3}-[0-9]{4}$
```

در این نسخه:

* `[0-9]{3}` یعنی سه رقم عددی پشت سر هم.
* `[0-9]{4}` یعنی چهار رقم عددی پشت سر هم.
* پرانتزها همچنان اختیاری‌اند چون با `\(?` و `\)?` مشخص شده‌اند.

---

#### آزمایش

```bash
echo "(555) 123-4567" | grep -E '^\(?[0-9]{3}\)? [0-9]{3}-[0-9]{4}$'
```

خروجی:

```
(555) 123-4567
```

---

```bash
echo "555 123-4567" | grep -E '^\(?[0-9]{3}\)? [0-9]{3}-[0-9]{4}$'
```

خروجی:

```
555 123-4567
```

---

```bash
echo "5555 123-4567" | grep -E '^\(?[0-9]{3}\)? [0-9]{3}-[0-9]{4}$'
```

خروجی ندارد، چون بخش اول چهار رقم دارد (باید دقیقاً سه رقم باشد).

این عبارت حالا به‌درستی شماره‌های تلفن را هم با پرانتز و هم بدون پرانتز می‌پذیرد، اما شماره‌های نادرست را رد می‌کند.

---

### کاربرد عملی عبارات منظم (Putting Regular Expressions to Work)

تا اینجا چند مثال تئوری دیدیم، اما regex زمانی واقعاً مفید است که آن را در اسکریپت‌ها و ابزارهای واقعی به‌کار بگیریم.
بیایید از `grep` برای **اعتبارسنجی لیست شماره تلفن‌ها** استفاده کنیم.

---

#### ایجاد لیست آزمایشی از شماره‌ها

با این دستور (که بعداً در فصل‌های بعدی دقیق‌تر یاد می‌گیریم) می‌توانیم فایل `phonelist.txt` شامل ده شمارهٔ تصادفی بسازیم:

```bash
for i in {1..10}; do echo "(${RANDOM:0:3}) ${RANDOM:0:3}-${RANDOM:0:4}" >> phonelist.txt; done
```

این دستور از متغیر `$RANDOM` استفاده می‌کند تا اعداد تصادفی بسازد.
اگر فایل را باز کنیم:

```bash
cat phonelist.txt
```

ممکن است خروجی مشابه زیر ببینیم:

```
(232) 298-2265
(624) 381-1078
(540) 126-1980
(874) 163-2885
(286) 254-2860
(292) 108-518
(129) 44-1379
(458) 273-1642
(686) 299-8268
(198) 307-2440
```

برخی از شماره‌ها ناقص یا خراب هستند، که برای تمرین عالی است.

---

#### یافتن شماره‌های نامعتبر با `grep`

برای پیدا کردن **خطوط ناصحیح** از گزینهٔ `-v` (یعنی نمایش معکوس نتایج) استفاده می‌کنیم:

```bash
grep -Ev '^\([0-9]{3}\) [0-9]{3}-[0-9]{4}$' phonelist.txt
```

خروجی:

```
(292) 108-518
(129) 44-1379
```

در این regex:

* `^` و `$` تضمین می‌کنند که کل خط بررسی شود (بدون نویسه‌های اضافی).
* پرانتزها اجباری‌اند (`\(` و `\)`)، بر خلاف مثال قبلی که اختیاری بودند.
* فقط شماره‌هایی که دقیقاً به شکل `(nnn) nnn-nnnn` هستند معتبرند.

و چون از `-v` استفاده کردیم، فقط **شماره‌های نامعتبر** در خروجی نمایش داده می‌شوند.

---

### یافتن نام فایل‌های مشکل‌دار با دستور `find`

(**Finding Ugly Filenames with find**)

دستور `find` از آزمونی به نام **`-regex`** پشتیبانی می‌کند که اجازه می‌دهد نتایج جستجو با استفاده از **عبارات منظم** فیلتر شوند.
اما باید یک نکتهٔ مهم را به خاطر داشت:
در حالی که `grep` **هر خطی را که شامل** یک تطبیق باشد چاپ می‌کند، دستور `find` تنها زمانی نتیجه‌ای را برمی‌گرداند که **کل مسیر فایل (pathname)** دقیقاً با عبارت منظم تطبیق داشته باشد.

برای نمونه، در مثال زیر می‌خواهیم تمام مسیرهایی را بیابیم که شامل نویسه‌هایی غیر از مجموعهٔ مجاز زیر باشند:

```
[-_./0-9a-zA-Z]
```

یعنی می‌خواهیم مسیرهایی را پیدا کنیم که مثلاً شامل فاصله یا نویسه‌های عجیب (مثل `!`, `@`, `*` و...) باشند:

```bash
find . -regex '.*[^-_./0-9a-zA-Z].*'
```

توضیح:

* چون `find` باید کل مسیر را تطبیق دهد، از `.*` در ابتدا و انتهای عبارت استفاده می‌کنیم تا هر تعداد نویسه قبل و بعد از الگوی اصلی مجاز باشد.
* در میان عبارت، از **عبارت منفی براکتی** `[ ^-_./0-9a-zA-Z ]` استفاده می‌کنیم تا بگوییم «هر چیزی غیر از این نویسه‌ها».

نتیجه این جستجو تمام فایل‌هایی را فهرست می‌کند که در نامشان فاصله یا نویسه‌های “زشت” دارند.

### جستجو با `locate`

(**Searching for Files with locate**)

برنامهٔ `locate` هم از regex پشتیبانی می‌کند —
هم در حالت **پایه‌ای** (`--regexp`) و هم در حالت **پیشرفته (Extended)** با گزینهٔ `--regex`.

مثلاً برای یافتن فایل‌هایی که در مسیرشان شامل `bin/bz` یا `bin/gz` یا `/bin/zip` هستند، می‌توانیم بنویسیم:

```bash
locate --regex 'bin/(bz|gz|zip)'
```

خروجی نمونه:

```
/bin/bzcat
/bin/bzcmp
/bin/bzdiff
/bin/bzegrep
/bin/bzexe
/bin/bzfgrep
/bin/bzgrep
/bin/bzip2
/bin/bzip2recover
/bin/bzless
/bin/bzmore
/bin/gzexe
/bin/gzip
/usr/bin/zip
/usr/bin/zipcloak
/usr/bin/zipgrep
/usr/bin/zipinfo
/usr/bin/zipnote
/usr/bin/zipsplit
```

با استفاده از **`|` (alternation)**، این دستور مسیرهایی را می‌یابد که هر یک از سه حالت یادشده را شامل شوند.

### جستجو در متن با `less` و `vim`

(**Searching for Text with less and vim**)

برنامه‌های `less` و `vim` هر دو از عبارات منظم برای جستجو در متن استفاده می‌کنند.
کافی است در آن‌ها کلید `/` را بزنید و سپس عبارت منظم خود را تایپ کنید.

مثلاً اگر فایل `phonelist.txt` را با `less` باز کنیم:

```bash
less phonelist.txt
```

و در پایین صفحه این عبارت را برای جستجو وارد کنیم:

```
/^\([0-9]{3}\) [0-9]{3}-[0-9]{4}$
```

`less` تمام شماره‌هایی را که با این الگو مطابقت دارند **هایلایت** می‌کند، و شماره‌های نامعتبر به‌راحتی دیده می‌شوند:

```
(232) 298-2265
(624) 381-1078
(540) 126-1980
(874) 163-2885
(286) 254-2860
(292) 108-518
(129) 44-1379
(458) 273-1642
(686) 299-8268
(198) 307-2440
```

در این مثال، مواردی مانند `(292) 108-518` و `(129) 44-1379` که با الگو تطبیق ندارند، بدون رنگ باقی می‌مانند.

---

در برنامهٔ `vim` نیز جستجو به همین شکل انجام می‌شود، با این تفاوت که `vim` از **عبارات منظم پایه‌ای (BRE)** استفاده می‌کند،
بنابراین بسیاری از متاکاراکترهایی که در ERE به‌صورت پیش‌فرض فعال‌اند، در vim باید با بک‌اسلش `\` نوشته شوند.

نمونهٔ جستجو در vim:

```
/([0-9]\{3\}) [0-9]\{3\}-[0-9]\{4\}
```

در اینجا `{3}` و `{4}` باید با `\{3\}` و `\{4\}` نوشته شوند، چون در BRE بدون بک‌اسلش معنی literal دارند.

اگر هایلایت فعال نباشد، می‌توان در حالت دستوری vim نوشت:

```
:hlsearch
```

> **نکته:**
> بعضی نسخه‌های لینوکس (مثل نسخهٔ پیش‌فرض Ubuntu) نسخه‌ای سبک از vim نصب می‌کنند که پشتیبانی کامل از هایلایت را ندارد.
> در این صورت، می‌توانید نسخهٔ کامل vim را با ابزار مدیریت بسته‌ها نصب کنید.

### جمع‌بندی (Summing Up)

در این فصل دیدیم که **عبارات منظم** چطور در ابزارهای مختلفی از لینوکس — از `grep` گرفته تا `find`، `locate`، `less` و `vim` — استفاده می‌شوند.
تقریباً در تمام ابزارهایی که با متن سروکار دارند، regex حضور دارد.

برای یافتن دیگر برنامه‌هایی که از regex پشتیبانی می‌کنند، می‌توان از `zgrep` برای جستجو در صفحات راهنمای فشرده استفاده کرد:

```bash
cd /usr/share/man/man1
zgrep -El 'regex|regular expression' *.gz
```

برنامهٔ `zgrep` در واقع نسخه‌ای از `grep` است که می‌تواند فایل‌های فشرده (مانند `.gz`) را مستقیماً بخواند.
با این دستور می‌توان دید که چه ابزارهایی در سیستم شما از regex استفاده می‌کنند — و معمولاً تعدادشان زیاد است.
